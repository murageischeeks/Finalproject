⚖️ Key Design Decisions
✅ MySQL
over PostgreSQL
OpenMRS uses a relational schema — reduces middleware data mapping complexity. Same DB engine means fewer type mismatches.
✅ Laravel
over Django / Node.js
Native RESTful routing, built-in middleware pipeline, Sanctum auth, Eloquent ORM, and mature queue system for async EMR sync.
✅ MVC Layered
over Microservices
Maintainable within an 8-month academic timeline. Single deployment unit. Service layer provides adequate separation.
✅ Write-Only Audit
No UPDATE / DELETE
audit_logs table uses raw DB::insert. No Eloquent mutators. No updated_at column. Tamper-evident by design.
✅ DB-Driven Triage Rules
over Hardcoded Logic
Weights stored in triage_rules table. Clinicians can adjust scoring without code changes. TriageClassificationService loads rules at runtime.
✅ SNOMED CT + LOINC
over Custom Codes
International clinical terminology standards. SubmissionTransformer maps symptoms to SNOMED CT and recovery status to LOINC for FHIR compliance.
🔧 Code-Level Improvements Found
🚨
CSRF Disabled
CRITICAL — Security Risk

VerifyCsrfToken.php passes all requests through without checking tokens. This disables CSRF protection entirely. Restore by extending Laravel's base VerifyCsrfToken and using $except to whitelist only API routes.

🚨
.env Credentials Exposed
CRITICAL — Gmail App Password in .env

Your .env file contains a hardcoded Gmail app password. If this repo is public, rotate immediately. Ensure .env is in .gitignore (currently only 14 bytes — suspiciously small).

🚨
DB Mismatch
CRITICAL — .env says PostgreSQL, docs say MySQL

DB_CONNECTION=pgsql in your .env, but the system is described as MySQL. The scopeByUrgency CASE expression works on both, but you should align documentation with actual config.

⚠️
Duplicate Triage Logic
MEDIUM — Two implementations exist

SubmissionController has a private classifyUrgency() with hardcoded weights + different thresholds (70/40) while TriageClassificationService uses DB-driven rules with thresholds (55/30). Only one should exist.

⚠️
Duplicate Controller Routes
MEDIUM — Two controllers handle follow-ups

Both Patient\FollowUpController and Patient\SubmissionController handle submission creation with different logic. SubmissionController uses hardcoded triage; FollowUpController uses the service. Consolidate into one.

⚠️
Orphan Patient Model
MEDIUM — Extends Authenticatable redundantly

Patient.php extends Authenticatable but User.php is used everywhere for patients. The Patient model appears unused — it's a leftover from early development. Remove or repurpose it.

💡
AuditLogService: Null User
LOW — Queued jobs have no auth context

auth()->id() returns null inside queued jobs. SyncSubmissionToEMR calls AuditLogService::log() which always inserts user_id = null. Pass user_id explicitly.

💡
Missing Doctor Model Integration
LOW — Doctor model is disconnected

The Doctor model exists with its own table but is never used. All doctor data lives in the users table with role='doctor'. Remove the model or migrate to a proper polymorphic setup.

💡
N+1 Query in Dashboard
LOW — Position calculation queries per appointment

PatientDashboardController runs a separate query for each appointment to compute queue position. Calculate positions in a single query using DB::raw() window functions or load all at once.

💡
No Rate Limiting on EMR API
LOW — Open simulated endpoint

The /api/emr/observations route has no auth and no rate limiting. In sandbox this is fine, but add throttle middleware and API key auth before any real deployment.

💡
FollowUpSubmission History: No Pagination
LOW — Patient history loads all records

Patient\FollowUpController::index() uses ->get() (loads all), while SubmissionController::index() uses ->paginate(10). Use pagination consistently.

💡
Admin Middleware Trace: No Role Guard
LOW — Any logged-in user can view traces

Route /admin/middleware-trace/{id} only requires auth — not role:admin. Any logged-in patient/doctor can view full audit traces. Add admin role check.
